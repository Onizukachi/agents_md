1. **Бизнес контекст для задачи**

Нужно дать возможность хранить и администрировать ставки НДС для партнёров (`Partner`) по аналогии с `operator_organizations`, но с отдельной моделью данных под партнёров. Для каждой ставки требуется период действия и дополнительный признак `vat_on_top`: если `true`, НДС начисляется сверху суммы; если `false`, НДС уже включён в сумму. По умолчанию `vat_on_top` должен быть `true`. Текущая ставка НДС должна отображаться на странице просмотра партнёра. Также нужна отдельная ActiveAdmin-админка для просмотра партнёрских ставок НДС с фильтрами по `partner_id` и по названию партнёра по подстроке.

2. **Ключевые решения и обоснования**

- Использовать отдельную таблицу ставок НДС партнёров, а не поля в `partners`, потому что у `operator_organizations` уже принят подход с историей ставок по периодам, и задача явно требует аналогичное поведение с датами действия.
- Хранить `vat_on_top` в той же таблице ставок НДС партнёров, потому что этот флаг относится к конкретной ставке и её трактовке, а не к партнёру в целом.
- Повторить базовую доменную логику `operator_organizations`: вложенное редактирование ставок, нормализация открытых периодов, запрет пересечений, вычисление ставки на дату. Это снижает когнитивную стоимость поддержки и делает поведение предсказуемым.
- Отдельную админку строить на модели ставок партнёров, а не на `Partner`, потому что фильтрация нужна именно по ставкам и связанному партнёру.
- До старта основной реализации нужно подтвердить спорные UX-детали: нужно ли редактирование ставок внутри формы партнёра, и как именно отображать `vat_on_top` в карточке/отдельной админке.

4. **Заметки для восстановления сессии**

- На момент создания плана в проекте уже реализованы ставки НДС для `OperatorOrganization` через модель `OperatorOrganizationVatRate`, nested-форму в `app/admin/operator_organizations.rb`, метод `vat_on(date)` и проверку пересечений периодов в модели.
- У `Partner` сейчас нет отдельной логики ставок НДС. Админка партнёра находится в `app/admin/partners.rb`, форма вынесена в `app/views/admin/partners/_form.html.haml`, поэтому при добавлении редактирования ставок почти наверняка потребуется менять и partial формы.
- В `.agents/tasks` до этой задачи не было файлов `task-*`; создан первый файл `task-1.md`.
- По локальным инструкциям любое изменение в `.agents/**/*` нужно зеркалить в `../agents_md/`, затем в зеркальном репозитории выполнить `git pull`, commit и push. Это нужно сделать сразу после создания данного файла.
- Критически важное архитектурное решение: новый флаг `vat_on_top` трактуется как атрибут конкретной ставки НДС партнёра, а не как атрибут партнёра.
- Неясности, требующие подтверждения до основной реализации:
  - нужно ли редактирование ставок прямо в форме партнёра, или только отображение в show и отдельная read-only админка ставок;
  - как именно показывать `vat_on_top` пользователю: отдельной колонкой/строкой или как часть текстовой подписи ставки;
  - нужна ли обратная совместимость с существующим полем `partners.nds`, если оно где-то ещё используется бизнесом.
- Если сессия прервётся, продолжать с проверки/синхронизации `../agents_md`, затем задать пользователю вопросы по UX и только после ответа переходить к миграции и коду.
