# LT-52595: Поддержать загрузку реестров операций HBPro

## 1. Бизнес-контекст

Бухгалтерии Level Travel нужно загружать реестры операций нового поставщика HBPro через существующую ActiveAdmin-страницу «Загрузка реестров операций». Реестр должен проходить тот же конвейер, что и недавно добавленный Ostrovok: предварительная фильтрация и сохранение файла, извлечение `OperatorOperation`, обогащение через Dynamics, сопоставление с заказом и создание либо возврат `OperatorPayment`.

Исходный пример: `/Users/hikaru/Downloads/HBPro от 20.07.2026.xlsx`. В книге один лист `Лист1`, первая строка содержит заголовки `№`, `Объект расчетов`, `Заказ клиента`, `Сумма`. В приложенном файле две положительные операции; примера отрицательной суммы нет.

Формат значимых полей:

- `Объект расчетов`: текст вида `Заказ клиента № ХП00260717-00679  от 17 июля 2026 г.`; из него нужно извлечь календарную дату операции;
- `Заказ клиента`: внешний номер заказа поставщика, в примере Excel хранит его как число (`2256085`);
- `Сумма`: строка вида `1 288,60(RUB)` с неразрывным пробелом, десятичной запятой и суффиксом валюты;
- положительная сумма означает оплату и записывается одновременно в `captured_amount` и `gross_captured_amount`;
- отрицательная сумма означает возврат, её абсолютное значение записывается одновременно в `refunded_amount` и `gross_refunded_amount`;
- валюта всегда `RUB`, курс — `1`.

Параметры интеграции:

- имя модуля и значение поставщика в интерфейсе: `Hbpro` (`Accounting::Hbpro`);
- `OPERATOR_ID = 78`;
- `SUPPLIER_ID = 104` для самих `OperatorOperation`;
- `DYNAMICS_ORGANIZATION_ID = 8` для получения booking ID через Dynamics; значение подтверждено командой Dynamics;
- `ORDER_SUPPLIER_ID = Accounting::Constants::Gortravel::SUPPLIER_ID` (`89`) для поиска заказа;
- `OPERATOR_PAYMENT_SUPPLIER_ID = Accounting::Constants::Gortravel::SUPPLIER_ID` (`89`) для создаваемых операторских платежей;
- `RELEASE_DATE = Date.new(2026, 7, 1)`;
- платежи HBPro должны иметь те же source flags, что Ostrovok: `deposit: true`, `payment_order: false`, `virtual: false`.

## 2. Ключевые решения и обоснования

1. **Встроить HBPro в существующий `Accounting::OperatorOperationsProcessor`.** Новый реестр использует последовательность `%i(extract enrich manage)`, чтобы не создавать отдельную оркестрацию и сохранить единое поведение загрузчика, логирования и обработки ошибок.

2. **Использовать `Hbpro` в именах Ruby-констант и selector ActiveAdmin.** В проекте уже существует namespace `OperatorAPI::Hbpro`; единое техническое написание избегает отдельного display mapping во view. Бизнес-написание `HBPro` сохраняется в логах и описании задачи.

3. **Разделить организацию операции и организации заказа/платежа.** `OperatorOperation` хранит новый `supplier_id = 104`, но заказ ищется и `OperatorPayment` создаётся с организацией Gortravel (`89`). Это повторяет подтверждённую бизнес-схему Ostrovok и требует явных `ORDER_SUPPLIER_ID` и `OPERATOR_PAYMENT_SUPPLIER_ID`.

4. **Матчить заказ через Dynamics, а не напрямую по `Заказ клиента`.** Значение `Заказ клиента` сохраняется как `supplier_order_id`, передаётся в Dynamics с подтверждённым `organization_id = 8`, из ответа берётся `booking.booking_id`, после чего заказ ищется по `operator_booking_id`, оператору `78` и supplier ID `89`. При нескольких совпадениях сохраняется текущая логика `DynamicsEnricher`: приоритет заказу с `booked_at`, затем самому раннему по `created_at`, затем существующий production fallback через Solr.

5. **Парсить дату явно из русского текста.** Нельзя полагаться на системную locale в `Date.parse`. Extractor должен выделить день, русское название месяца и год, преобразовать месяц через локальную таблицу и создать дату. Некорректное или отсутствующее значение должно приводить к понятной ошибке строки/операции, а не к молчаливой подстановке текущей даты.

6. **Нормализовать денежную строку до `BigDecimal`.** Перед преобразованием нужно удалить обычные и неразрывные пробелы, суффикс `(RUB)`, заменить десятичную запятую точкой и сохранить знак. После определения типа операции в суммы записывается абсолютное значение. Нулевая либо нераспознанная сумма не должна создавать оплату или возврат.

7. **Заполнять одновременно обычную и gross-сумму.** Для оплаты `captured_amount == gross_captured_amount`, для возврата `refunded_amount == gross_refunded_amount`. Благодаря этому стандартная логика capture/refund сможет работать с HBPro без нового supplier-specific переключения сумм, а данные операции останутся полными.

8. **Дедуплицировать по имени файла и номеру строки, как в `Accounting::Gortravel::Extractor`.** `external_id` формируется как `"#{filename}::#{row_number}"`. Повторная обработка файла с тем же именем не создаст дубли, а две одинаковые по дате, заказу и сумме операции в разных строках не схлопнутся. Осознанный компромисс: переименованный файл считается новым и может повторно создать те же операции.

9. **Сохранить идентификаторы Excel строками на этапе `FileProcessor`.** Уже существующая принудительная запись строк защищает `Заказ клиента` от scientific notation и потери точности. Extractor дополнительно должен корректно нормализовать как строковое, так и числовое значение supplier order ID.

10. **Сделать выбор supplier ID для заказа и платежа расширяемым.** Текущий код содержит специальную ветку только для Ostrovok. При добавлении HBPro нужно либо обобщить выбор через supplier constants, либо аккуратно добавить HBPro во все соответствующие ветки; тестами обязательно зафиксировать отсутствие изменений для остальных поставщиков.

11. **Не добавлять gem и не менять схему БД.** Формат можно разобрать имеющимися Roo, ActiveSupport и стандартными классами даты/чисел; новые persisted-поля не требуются.

- Рабочая ветка на момент составления плана: `hotfix/LT-52595-operator-operations-loader-support-hbpro_dev`, основана на актуальном `develop` (`774063e4d7`).
- В рабочем дереве до создания этого артефакта не было изменений.
- Реализованы константы, регистрация в загрузчике, `Accounting::Hbpro::{FileUploader,Extractor,Enricher,RegisterValueParser}` и supplier mapping платежей. В selector ActiveAdmin используется техническое имя `Hbpro` без отдельного display mapping.
- Добавлены specs parser/extractor/enricher/FileProcessor и HBPro capture/refund/cancel, а также factory traits.
- В репозитории уже присутствует реализация Ostrovok из коммита `b8853ac65518c3ba7ca872e91fff63f1ea120681`: её extractor, enricher, file uploader и model specs являются основным образцом.
- Приложенный файл успешно открыт через Roo без запуска Rails. Он содержит один лист, заголовок и две положительные строки; отрицательные, нулевые и повреждённые значения необходимо синтезировать в тестах.

Добавлены HBPro regression specs для ошибки API, пустого booking ID и отсутствующего заказа.
- Проверено, что `Accounting::OperationsLoader::XLSXReader` преобразует числовой `Заказ клиента` в строку до применения фильтра; regression spec фиксирует совпадение числовой Excel-ячейки `2256085` со строковым параметром фильтра.
- Точечный Docker RSpec после исправлений: 13 примеров, 0 ошибок. Расширенный набор `spec/services/accounting` плюс существующий и новый `OperatorOperation` specs: 119 примеров, 0 ошибок.
- TeamCity-compatible gate не запускался: `PROTO_REPO_TOKEN` отсутствует, `gh` не авторизован. Host Bundler также не является CI-equivalent и ранее был заблокирован отсутствующим checkout git-gem `markerb`.

### Критически важные принятые решения

- Namespace и значение в selector ActiveAdmin — `Hbpro`; бизнес-написание в логах — `HBPro`.
- `OperatorOperation`: operator `78`, supplier `104`.
- Dynamics: подтверждённая organization `8`.
- Поиск заказа и `OperatorPayment`: supplier `89` (`Gortravel::SUPPLIER_ID`).
- Дата начала обработки/поиска возвратов: `01.07.2026`.
- Платежи депозитные, как у Ostrovok.
- Дата берётся только из текста `Объект расчетов`; системная locale не должна влиять на парсинг русских месяцев.
- Знак `Сумма` определяет тип операции. В persisted amount columns сохраняются абсолютные значения; для capture и refund заполняются одновременно обычная и gross-колонки.
- Валюта всегда `RUB`, курс всегда `1`.
- `external_id` формируется из имени файла и номера строки по аналогии с Gortravel. Это сохраняет кратность одинаковых операций и дедуплицирует повторную обработку того же имени файла, но не защищает от загрузки того же содержимого после переименования.
- Новые gems и migration не нужны.
