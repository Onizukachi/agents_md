# LT-52624: Добавить обработку партнёрских реестров МКБ

## 1. Бизнес-контекст для задачи

Партнёр МКБ (`Partner` с `id = 1207`) передаёт Excel-реестры операций по партнёрской программе. Эти файлы нужно загружать через существующую ActiveAdmin-страницу «Загрузка реестра партнёров» и преобразовывать в унифицированные `PartnerOperation`, чтобы сотрудники могли:

- связать строку банковского реестра с внутренним заказом и платежом Level.Travel;
- сохранить RRN, дату операции, сумму комиссии и исходные данные строки;
- рассчитать `gross_cost` с учётом `Partner#cost_ratio`;
- проверить результат в ActiveAdmin и выгрузить операции в Excel;
- увидеть атомарную ошибку обработки на уровне всего файла, не сохраняя частичный результат.

Приложенный пример `Июнь 2026 МКБ - реестр (АС).xlsx` содержит один лист. Перед табличными данными расположена служебная шапка реестра, поэтому фактические заголовки находятся в Excel-строке 4, а не в первой строке. После строк операций присутствует пустая строка и строка итогов. Значимые для импорта колонки:

- `Дата и время Расходной операции / Операции возврата (ДД.ММ.ГГГГ; 00:00)` — дата и время операции;
- `Сумма Расходной операции / Операции возврата, руб.` — сумма операции для резервного сопоставления платежа;
- `Комиссия, руб.` — значение `PartnerOperation#cost`, включая отрицательный знак для возврата;
- `RRN номер` — RRN для поиска и привязки `Payment`;
- `Order_ID` — внутренний `orders.id` Level.Travel для прямой привязки заказа.

В примере также есть карточные и merchant-атрибуты, `operator_name` и `order_status`. Они не участвуют в вычислениях, но должны сохраниться в `PartnerOperation#data`, если reader возвращает их в строке. Production-файлы будут приходить без пароля. Пароль приложенного локального примера использовался только для анализа формата и не должен попадать в код, fixtures, task-артефакт или git-историю.

## 2. Ключевые решения и обоснования

1. **Добавить `PartnerRegistries::Rules::Mkb < Base`.** Новый обработчик должен использовать существующий общий workflow `RegistryFile -> ProcessFileWorker -> Processor -> PartnerOperation`, без отдельного контроллера, worker или параллельной модели данных.

2. **Не зашивать `partner_id = 1207` в правило.** Обработчик выбирается через существующее поле `partners.registry_processor`. Для партнёра 1207 нужно установить значение `Mkb` через карточку партнёра и проверить `cost_ratio`; сам класс остаётся переиспользуемым и следует текущей архитектуре партнёрских правил.

3. **Не реализовывать расшифровку файлов и не добавлять зависимость для неё.** Пользователь подтвердил, что production-реестры загружаются без пароля. Локальный пример нужно расшифровать только во временную директорию для анализа/ручной проверки; расшифрованную копию и пароль нельзя коммитить.

4. **Искать строку заголовков по сигнатуре, а не фиксировать номер 4.** В `Spreadsheet::Reader` нужно добавить опциональный способ найти Excel-заголовок по уникальным колонкам МКБ, сохранив текущий путь чтения для остальных реестров. Это переживёт изменение высоты или количества строк служебной шапки и не создаст MKB-специфичный парсинг внутри общего processor.

5. **Передавать настройки чтения декларативно из rule-класса.** `Mkb` должен объявить `READ_SCHEMA` и сигнатуру поиска заголовка; `PartnerRegistries::Processor` передаёт поддерживаемые reader options в `Spreadsheet::Reader`. Для классов без этих настроек поведение остаётся прежним.

6. **Использовать точные заголовки из реестра и типизированную схему.** Дата читается как `DateTime`, сумма операции и комиссия — как числовые значения, RRN — как строка, `Order_ID` — как целое либо безопасно нормализуемая строка. Формульная ячейка комиссии должна читаться по сохранённому Excel-значению, как в приложенном файле.

7. **Связывать заказ напрямую по `Order_ID`.** Если указанный заказ существует, его `id` записывается в `PartnerOperation#order_id`, независимо от того, удалось ли найти платёж. При пустом или несуществующем `Order_ID` допустим существующий fallback через найденный по RRN/логам платёж, аналогично правилам ПСБ и ВТБ.

8. **Платёж продолжать искать общей логикой `Base`.** Приоритет имеет точное совпадение `Payment#rrn`; если оно отсутствует, используется существующий fallback по дате и абсолютной сумме операции в логах Uniteller. Найденный платёж получает `partner_operation_id`.

9. **Считать `cost` из `Комиссия, руб.` без изменения знака.** Положительная комиссия остаётся положительной, комиссия возврата — отрицательной. `gross_cost` рассчитывается существующим `Base` как `cost * partner.applied_cost_ratio(false)`; при отсутствии настроенного коэффициента применяется текущий fallback `1.0`.

10. **Не создавать операции из служебных и итоговых строк.** Строка итогов не содержит RRN и даты, поэтому должна быть пропущена общим guard в `Base`. Отсутствующие обязательные заголовки или неразбираемые обязательные значения должны давать понятную ошибку обработки файла, а не статус `processed` с незаметно пустым результатом.

11. **Не использовать реальный файл из `~/Downloads` в автотестах.** Добавить обезличенный незашифрованный fixture с тем же устройством листа: служебная шапка, найденный не в первой строке header, положительная операция, возврат, пустая строка и итоги. Это делает тесты воспроизводимыми в Docker/CI и не сохраняет реквизиты или карточные данные из исходного документа.

12. **Не добавлять gem и не менять схему БД.** Roo 2.7.1, текущий `Spreadsheet::Reader`, существующие поля партнёра и `PartnerOperation` достаточны. Настройка `registry_processor` является операционной конфигурацией существующего партнёра, а не причиной для новой колонки.

## 3. Структурированный список задач

- [ ] Шаг 2: Добавить regression specs для текущего `Spreadsheet::Reader`, фиксирующие неизменное чтение существующих Excel/CSV-файлов при отсутствии новых опций.
- [ ] Шаг 3: Расширить `Spreadsheet::Reader` опциональным поиском строки Excel-заголовков по набору уникальных названий колонок; нормализовать пробелы в найденных заголовках тем же способом, что и в текущем `process_row`.

### Текущее состояние проекта и уже выполненная работа

- Рабочая ветка: `feature/LT-52624-support-mkb-partner-registries`.
- Базовый commit на момент анализа: `d854c74ed0`, совпадает с текущим `develop`/`origin/develop`.
- До создания этого task-артефакта рабочее дерево было чистым; прикладной код для МКБ ещё не менялся.
- Изучена полная цепочка существующего импорта: `app/admin/registry_files.rb`, `RegistryFile`, `PartnerRegistries::ProcessFileWorker`, `PartnerRegistries::Processor`, `Spreadsheet::Reader`, `PartnerRegistries::Rules::Base`, партнёрские rules и ActiveAdmin для `PartnerOperation`.
- Приложенный файл был расшифрован только во временной директории для анализа. Он содержит один лист, служебные строки 1–3, заголовок в строке 4, одиннадцать строк данных, пустую строку и итоговую строку. В репозиторий копия файла не добавлена.
- В приложенном примере даты являются нативными Excel date/time, суммы операций — числовыми значениями, комиссия — формулой с сохранённым числовым результатом, `RRN номер` — строкой, `Order_ID` — числом.
- Все строки реального примера положительные. Отрицательный возврат и ошибочные значения необходимо синтезировать только в обезличенном test fixture.
- Текущий `Spreadsheet::Reader` для Excel предполагает заголовок в начале листа (`parse(headers: true)`), поэтому напрямую этот формат МКБ разобран не будет. Это первый технический блокер реализации.
- Специализированных specs для `PartnerRegistries::Processor` и большинства rules сейчас нет; есть только общие specs `Spreadsheet::Reader` и model specs отображения `PartnerOperation`.
- Существующие ПСБ/ВТБ rules являются ближайшим примером прямой связи по order id; `Base` уже реализует RRN/fallback matching, нормализацию времени, `cost_ratio`, `gross_cost`, сохранение source data и привязку платежа.

### Критически важные принятые решения

- Production-файлы МКБ не зашифрованы; парольная логика не входит в задачу.
- Партнёр МКБ имеет `id = 1207`, но ID не должен быть зашит в rule-класс.
- Значение processor для карточки партнёра — `Mkb`, класс — `PartnerRegistries::Rules::Mkb`.
- `cost` берётся из `Комиссия, руб.` с исходным знаком; `gross_cost = cost * cost_ratio`.
- `Order_ID` является внутренним ID заказа Level.Travel и используется для прямой связи.
- `RRN номер` используется для поиска конкретного платежа и его связи с `PartnerOperation`.
- Сумма расходной операции/возврата передаётся в общую fallback-логику поиска платежа; для возврата сопоставление идёт по абсолютному значению.
- Строка заголовков должна находиться по сигнатуре, а не по жёсткому номеру строки.
- Реальный файл, его пароль, реквизиты и карточные данные нельзя коммитить; тестовый fixture должен быть обезличен.
- Новые зависимости, схема БД и PAPI-документация для реализации не требуются.

### Точка, с которой нужно продолжить, если сессия прервётся

Начать с проверки `git status` и содержимого этого task-файла. Затем выполнить шаг 1: создать обезличенный незашифрованный fixture, отражающий структуру листа МКБ. После этого зафиксировать regression specs текущего reader и реализовать опциональный header search до создания `PartnerRegistries::Rules::Mkb`. Не начинать с MKB rule поверх неизменённого reader: он не увидит заголовки приложенного формата из-за служебной шапки. После любых изменений `AGENTS.md` или `.agents/` повторно выполнить обязательный sync через `.agents/skills/leveltravel-agents-sync`.
