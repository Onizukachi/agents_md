# Оптимизация генерации купонов промо-кампаний

## 1. Бизнес контекст для задачи

- Генерация больших промо-кампаний с количеством купонов от `10_000` до `200_000` сейчас выполняется одной Sidekiq-джобой и может работать слишком долго, зависать или падать по таймаутам/ошибкам.
- Бизнесу нужна устойчивая генерация больших объёмов купонов без ручного вмешательства и без блокировки остальных фоновых задач.
- Уже созданные купоны при частичных сбоях должны сохраняться, чтобы следующая операция могла догенерировать только недостающее количество.
- В системе уже допускается повторный запуск генерации после изменения `promo_campaign.coupons_count`, поэтому новая реализация должна корректно работать и для частичных повторных запусков, и для увеличения планового количества купонов.

## 2. Ключевые решения и обоснования

- Генерацию нужно разбить на batch jobs, а не на `1 job = 1 coupon`.
  Это даёт параллельность без избыточной нагрузки на Redis/Sidekiq и без слишком частой синхронизации состояния.
- Главный воркер остаётся coordinator-воркером.
  Он фиксирует snapshot целевого состояния в `PromoCampaignOperation.metadata`, рассчитывает дельту и раскладывает batch jobs.
- Каждая новая операция генерации работает только с дельтой:
  `remaining_coupons_count = target_coupons_count - existing_coupons_count`.
  Это поддерживает оба сценария:
  - повторный запуск после частичного падения;
  - повторный запуск после увеличения `coupons_count`.
- Если `remaining_coupons_count <= 0`, операция должна завершаться без запуска batch jobs как `success`/`no-op` сценарий.
  Это самый простой и предсказуемый вариант, не требующий отдельного статуса операции.
- Каждый batch создаёт фиксированное количество купонов внутри одной DB-транзакции.
  Это исключает частично созданные батчи и защищает от переизбытка купонов при Sidekiq retry.
- Синхронизация batch jobs должна вестись только через `PromoCampaignOperation.metadata`.
  Не добавлять Redis-координацию, Sidekiq Batch или отдельные служебные таблицы.
- `coupons.pin` должен иметь уникальный индекс на уровне БД.
  При параллельной генерации нельзя полагаться только на `Coupon.exists?` и Rails validation.
- Генерация `pin` должна ретраиться на конфликте уникальности.
  Это позволит убрать хрупкость при параллельной вставке и снизит риск гонок.
- Очередь генерации нужно поднять выше `low`, но не использовать `high`.
  Предпочтительный вариант для реализации: `normal`, чтобы ускорить выполнение, не отбирая ресурсы у критичных джоб.
- Избегать оверинжиниринга:
  не добавлять новую сложную оркестрацию, не строить многоуровневую систему восстановления, не вводить новый жизненный цикл сверх `processing/success/failed`.

## 4. Заметки для восстановления сессии

### Текущее состояние проекта и уже выполненная работа

- Исследована текущая реализация генерации:
  - `app/workers/promo_campaigns/coupon_generator_worker.rb` сейчас синхронно создаёт все купоны в одном `perform`.
  - `app/services/promo_campaigns/coupon_creator.rb` генерирует `pin` через цикл с `Coupon.exists?(pin: pin)`.
  - `app/models/promo_campaign_operation.rb` уже хранит `metadata` и `generated_coupons_count`.
  - `app/models/promo_campaign.rb` уже создаёт `PromoCampaignOperation` через `start_generation!`.
  - `app/admin/promo_campaigns.rb` запускает воркер после `start_generation!`.
- Проверено текущее состояние индексов:
  - на `coupons.pin` уже есть обычный индекс;
  - уникального индекса нет.
- Уточнено у пользователя:
  - исторических дублей `pin` в базе нет;
  - максимальный целевой объём генерации до `200_000` купонов;
  - допустим сценарий повторного запуска генерации после изменения `coupons_count`;
  - вариант `no-op`, когда купонов уже достаточно, согласован.

### Критически важные принятые решения

- Не использовать `1 job = 1 coupon`.
- Использовать `1 job = 1 batch`.
- Не использовать отдельную инфраструктуру синхронизации, только `PromoCampaignOperation.metadata`.
- Не пытаться “долечивать” старую операцию; каждая генерация создаёт новую `PromoCampaignOperation`.
- Каждая операция считает дельту только один раз в начале по snapshot:
  `target_coupons_count`, `existing_coupons_count`, `remaining_coupons_count`.
- Если `remaining_coupons_count <= 0`, операция завершается без batch jobs.
- Если часть батчей упала после исчерпания retry, операция становится `failed`, но уже созданные купоны сохраняются.
- Следующая операция должна догенерировать только нехватающее количество купонов.
- Очередь генерации поднимается выше `low`; базовый кандидат для реализации: `normal`.

### Точка, с которой нужно продолжить, если сессия прервётся

- Следующее действие: согласовать оставшиеся прикладные параметры реализации, затем переходить к коду.
- Перед кодом нужно уточнить:
  - точный `batch_size`;
  - нужен ли отдельный список `failed_batches` в `metadata` или достаточно счётчиков и последней ошибки;
  - какое итоговое состояние должно быть у `no-op` операции: `success` с нулём сгенерированных купонов.
- После согласования параметров:
  1. внести миграцию на unique index для `coupons.pin`;
  2. реализовать batch-воркер и coordinator;
  3. обновить тесты;
  4. прогнать релевантный RSpec.
