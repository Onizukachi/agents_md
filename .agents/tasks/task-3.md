# История ставок НДС для operator_organizations

## 1. Бизнес-контекст задачи

Для юридического лица туроператора из таблицы `operator_organizations` нужно хранить не одно текущее значение НДС, а историю ставок с периодами действия.

Ставка должна выбираться по дате создания заказа `receipt.order.created_at`. Менеджер должен иметь возможность редактировать историю ставок прямо в текущей форме `operator_organization` в ActiveAdmin без перехода в отдельный ресурс.

Если у ставки не указана дата окончания, она действует бессрочно до момента, пока не будет заведена следующая ставка. При создании новой ставки предыдущую открытую ставку нужно автоматически закрывать датой `starts_at - 1 день`.

Изменение должно работать одинаково для обоих фискальных провайдеров, которые используют данные поставщика: `Uniteller` и `CheckOnline`.

Текущее поле `operator_organizations.vat` пока остается в базе для безопасного перехода, но должно быть убрано из формы и отображения в админке. Перенос существующих значений в новую таблицу будет выполняться отдельной rake-задачей: для каждой организации с заполненным `vat` создается бессрочная запись со `starts_at = 2026-01-01`.

## 2. Ключевые решения и обоснования

1. Историю ставок хранить в отдельной таблице `operator_organization_vat_rates`, а не расширять `operator_organizations`.
   Обоснование: требуется несколько периодов для одной организации, валидация непересекающихся интервалов и безопасная миграция без потери истории.

2. Дату действия ставки определять по `receipt.order.created_at.to_date`.
   Обоснование: это явно согласовано для бизнес-логики, и сравнение по дню убирает неоднозначность на границе суток.

3. Настройку ставок сделать через nested form в `app/admin/operator_organizations.rb`.
   Обоснование: пользователь просил оставить управление в текущей форме, а в проекте уже используются `f.has_many` и `accepts_nested_attributes_for`.

4. Пересечения периодов для одной организации запрещать полностью.
   Обоснование: это исключает неявный выбор ставки и упрощает поведение фискализации.

5. При сохранении новой ставки с `starts_at = X` автоматически закрывать предыдущую открытую ставку датой `X - 1 день`.
   Обоснование: это соответствует согласованному поведению "действует бесконечно до следующей ставки" и снижает ручную нагрузку на менеджера.

6. Допустимые значения ставки ограничить целыми числами от 1 до 100.
   Обоснование: это явно подтверждено пользователем; вариант "без НДС" в этой задаче не поддерживается.

7. Старое поле `operator_organizations.vat` не удалять в рамках этой задачи.
   Обоснование: оно нужно как временный fallback для миграционного периода и rake-переноса, но не должно больше редактироваться через админку.

8. Rake-задачу миграции сделать отдельно от схемной миграции.
   Обоснование: пользователь явно задал способ переноса данных и дату начала исторических записей `2026-01-01`.

9. Перед реализацией нужно уточнить маппинг значения ставки в формат `CheckOnline`.
   Обоснование: в текущем коде `CheckOnline` использует только `TaxId = 4`, а пользовательская модель ставок теперь допускает целые значения `1..100`; без явного правила нельзя корректно завершить интеграцию.
