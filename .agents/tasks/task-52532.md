# LT-52532: Исправить ложную отправку уведомлений о документах

## 1. Бизнес-контекст

Level Travel синхронизирует PDF-документы заказа от туроператоров и уведомляет внешних партнёров/субагентов об изменении документов. Сейчас признаком изменения служит `Attachment#source_hash`.

В некоторых ответах туроператора набор страниц документа остаётся прежним, но меняется их порядок. Из-за этого меняется `source_hash`, attachment перезаписывается и партнёру уходит событие `documents`, хотя содержательно новых страниц нет. Такие уведомления ложные и создают лишние действия на стороне субагента.

Требуемое поведение: хранить `content_hash`, зависящий от мультимножества страниц PDF, но не от их порядка:

1. получить все страницы PDF;
2. вычислить детерминированный hash каждой страницы;
3. отсортировать page hashes;
4. сконкатенировать их, сохранив повторы;
5. вычислить итоговый hash.

Перестановка `AB -> BA` должна сохранить `content_hash` и подавить только уведомление. Сам файл и `source_hash` должны продолжить синхронизироваться по текущим правилам. Изменение, добавление или удаление страницы должно изменить `content_hash` и привести к уведомлению.

## 2. Как это работает сейчас

### Цепочка отправки уведомления

- `Attachment` регистрирует `after_save :make_notify`: `app/models/attachment.rb:24-25`.
- `make_notify` пропускает не-`Order`, заказ без партнёра, неподходящее состояние, партнёра без внешних уведомлений и заказ без webhook URL: `app/models/attachment.rb:80-88`.
- Единственный признак изменения документа — `saved_change_to_attribute?('source_hash')`: `app/models/attachment.rb:93`.
- При выполнении условия создаётся несохранённый `OrderLog` с `kind: 'action_track'` и `action_code: 'attachment_create'`, после чего синхронно вызывается `Order::ExternalNotification::Sender`: `app/models/attachment.rb:94-102`.
- Sender повторно проверяет допустимость партнёра и события, создаёт `ExternalNotification` и ставит `OrderExternalNotification` на отправку: `app/services/order/external_notification/sender.rb:14-37,52-71`.
- Событие attachment разрешается по сообщению `Загружен...` и `kind == 'action_track'`: `app/services/order/external_notification/sender.rb:59-63`.

### Два пути синхронизации документов

1. `OrderSync::Services::UpdateOrderAttachments`

   - Сравнивает только `previous_attachment.source_hash` и `document.signature`: `app/apis/order_sync/services/update_order_attachments.rb:6-12`.
   - При различии удаляет старый attachment и создаёт новый в транзакции: `app/apis/order_sync/services/update_order_attachments.rb:24-33`.
   - Новый attachment получает входной PDF и `source_hash: document.signature`: `app/apis/order_sync/services/update_order_attachments.rb:36-48`.
   - На новом record dirty change всегда выглядит как `nil -> source_hash`, поэтому callback отправляет уведомление.

2. `OrderSync::Services::UpdateBookingInfo#update_documents`

   - Обновляет существующий attachment: title, file и `source_hash`, затем вызывает `save!`: `app/apis/order_sync/services/update_booking_info.rb:133-147`.
   - В этом пути сравнение старого и нового persisted `content_hash` возможно на одной записи, но для legacy-строк с `content_hash = nil` всё равно нужна отдельная стратегия.

`source_hash` не является единым hash бинарного содержимого PDF: разные adapters строят signature из списков документов, идентификаторов, текста страниц или других данных. Порядок входных документов/страниц влияет на него. Поэтому менять семантику `source_hash` нельзя; `content_hash` должен быть отдельным признаком именно для решения об уведомлении.

### Текущее тестовое покрытие

- Прямого model spec для `Attachment#make_notify` нет.
- `spec/services/order/external_notification_table_spec.rb:75-113` проверяет только положительные сценарии create/update по `source_hash`.
- `spec/apis/order_sync/services/update_order_attachments_spec.rb` проверяет replace/skip по `source_hash`, но не перестановку страниц и не уведомления.
- `spec/apis/order_sync/services/update_booking_info_spec.rb:262-309` проверяет обновление полей, но не содержимое PDF и не уведомления.
- `spec/services/order/external_attachment_spec.rb` проверяет payload/delivery уже созданного уведомления, а не условие его создания.
- `spec/fixtures/attachment.pdf` содержит две страницы, но для `CombinePDF` требует `allow_optional_content: true`.

### Обнаруженные технические риски

- Простая замена `saved_change_to_attribute?('source_hash')` на `saved_change_to_attribute?('content_hash')` не работает для `destroy + create`: у нового record `content_hash` также меняется из `nil` в значение.
- Наивный page hash через `(CombinePDF.new << page).to_pdf` недетерминирован: `CombinePDF 1.0.16` добавляет текущие `CreationDate`/`ModDate`, а merge/reparse перенумеровывает PDF objects/references. Удаления дат недостаточно.
- Hash только по `PDF::Reader::Page#text` не учитывает изображения, формы, шрифты и часть визуального содержимого; это может скрыть реальное изменение.
- Рендер страницы в pixels даёт понятную семантику визуального равенства, но требует закреплённой runtime-зависимости и параметров renderer. `poppler/pdftoppm` сейчас не объявлен в Dockerfiles проекта.
- Вычисление hash внутри model callback может читать файл из S3 через `Attachment#open_file`, замедлять `save`/транзакцию и ломать синхронизацию на malformed/encrypted PDF.
- Все существующие attachments после добавления колонки будут иметь `content_hash = nil`. Массовый backfill потребует чтения файлов из S3 и парсинга PDF; выполнять его внутри schema migration нельзя.
- `AttachmentUploader` после store повторно вызывает `model.save!`: `app/uploaders/attachment_uploader.rb:3-4,60-69`. Новая логика не должна создавать дубли уведомлений на втором save.
- Сейчас константа `NOTIFICABLE_ORDER_STATES` включает `hotel_confirmed`, а ранний guard в `make_notify` — нет. Это существующая несогласованность и не входит в LT-52532.

## 3. Ключевые решения и обоснования

1. **`source_hash` оставить без изменения.** Он продолжает управлять обнаружением обновления у поставщика и самим обновлением attachment. Новый `content_hash` используется только как содержательный признак для уведомления.

2. **Сравнивать старый и новый content hash до уничтожения старого attachment.** Для `UpdateOrderAttachments` previous hash должен быть прочитан из колонки либо вычислен из старого файла до `destroy`; результат сравнения нужно передать новому attachment как transient notification context. Dirty tracking нового record сам по себе недостаточен.

3. **Сохранить уведомление при первичном создании.** Если предыдущего документа действительно нет, создание PDF остаётся событием `documents`.

4. **Не выполнять массовый backfill в migration.** Колонка добавляется nullable и без индекса. Для legacy attachment sync-path должен лениво получить hash старого файла до его замены. После успешного сохранения новый hash окажется в БД. Это исключает полный обход S3 при деплое.

5. **Вычислять hash из уже доступных данных до DB-транзакции.** Для нового документа использовать `document.file_content`/CarrierWave cached file; persisted `content_hash` старого документа читать из БД. Удалённый файл открывать только для legacy fallback, когда hash отсутствует.

6. **Hash-алгоритм версионировать и использовать SHA-256.** Page hashes имеют фиксированную hex-длину, поэтому их можно безопасно сортировать и конкатенировать без delimiter; одинаковые страницы не дедуплицировать. Версия алгоритма должна быть константой/частью контракта сервиса, чтобы будущая смена реализации не выглядела как изменение документа.

7. **До продуктовой интеграции доказать детерминизм представления страницы.** Предпочтителен pure-Ruby вариант без нового gem/runtime dependency, но только если corpus/specs подтверждают, что canonical representation включает page content и связанные resources, не зависит от metadata, object IDs, времени и позиции страницы. Если это невозможно надёжно обеспечить, использовать фиксированный rasterization contract и отдельно добавить/pin renderer в runtime image. Наивный `CombinePDF#to_pdf` и text-only hash запрещены.

8. **Ошибки hash не должны ломать синхронизацию документа.** Malformed/encrypted/неподдержанный PDF нужно отправить в Sentry и применить fail-open fallback: сохранить текущее поведение по `source_hash` и отправить уведомление при его изменении. Это безопаснее, чем молча скрыть реальное изменение.

9. **Для non-PDF сохранить текущее поведение.** Если callback срабатывает для другого формата, использовать hash всего файла при доступном содержимом, иначе fallback на `source_hash`. LT-52532 не должен отменить существующее уведомление при замене PDF на DOC/DOCX/image.

10. **Логику вычисления вынести в namespaced service.** Например, `Attachments::ContentHasher`; модель должна хранить hash и принимать решение об уведомлении, но не содержать сложный PDF parser/renderer workflow.

### Таблица ожидаемого поведения

| Предыдущий документ | `source_hash` | Страницы | Результат |
|---|---|---|---|
| отсутствует | новый | новый PDF | сохранить и уведомить |
| есть | изменился | те же страницы, другой порядок | сохранить/заменить файл, не уведомлять |
| есть | изменился | изменена хотя бы одна страница | сохранить/заменить файл, уведомить |
| есть | изменился | добавлена/удалена страница | сохранить/заменить файл, уведомить |
| есть | тот же | те же страницы | текущее sync-поведение `UNCHANGED`, не уведомлять |
| legacy hash отсутствует | изменился | старый файл доступен | вычислить old hash лениво и сравнить |
| legacy hash отсутствует | изменился | старый файл/парсинг недоступны | сохранить, сообщить ошибку в Sentry, уведомить по `source_hash` |

## 5. Заметки для восстановления сессии

### Текущее состояние

- Выполнен только анализ; продуктовый код, migration, schema и specs не менялись.
- Создан этот артефакт `.agents/tasks/task-52532.md`.
- Рабочая ветка: `feature/LT-52532-subagent-prevent-false-document-notifications-from-being-sent`.
- На момент анализа рабочее дерево не содержало видимых изменений.

### Критически важные решения

- Нельзя ограничиться заменой dirty-check с `source_hash` на `content_hash`.
- В `UpdateOrderAttachments` старый record удаляется; old hash нужно получить до `destroy` и передать в notification context нового record.
- Перестановка страниц подавляет только webhook, но не обновление файла/source hash.
- Legacy rows обрабатываются лениво; массового S3 backfill внутри migration нет.
- Ошибка hash — fail-open по текущему `source_hash` с Sentry, а не падение синхронизации и не молчаливое подавление.
- Наивная сериализация страницы CombinePDF и text-only hashing не соответствуют контракту.
- Алгоритм должен быть доказан тестами на стабильность во времени, независимость от object IDs/metadata/порядка и чувствительность к изображениям/resources.

### Точка продолжения

Начать с шагов 1-3: подготовить AB/BA/AC/ABB fixtures и executable contract specs для hasher. Только после прохождения детерминизма выбрать canonical representation и переходить к migration/model/sync integration. Для migration обязательно использовать `leveltravel-migrations`, для запуска тестов — `leveltravel-tests`.
